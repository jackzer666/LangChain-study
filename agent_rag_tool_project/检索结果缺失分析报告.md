# agent_rag_tool_project 检索结果缺失分析报告

生成时间：2026-05-19

## 结论摘要

当前项目的“检索结果缺失”不是单点问题，更像是索引构建、分片策略、混合召回、Agent 调用和评估口径共同造成的结果。最明显的风险是：`chunks.jsonl` 当前有 70 条记录，但 Chroma `embeddings` 表有 140 条记录，说明向量库至少重复写入过一轮；同时 BM25 使用的是 `chunks.jsonl`，向量检索使用的是 Chroma，两套索引数量和元数据不一致，会导致混合检索排序不稳定、重复内容占用 top-k、相关结果被挤出。

项目本身是“小知识库文件内容检索”，目前引入 Agent、Middleware、二次总结链路和 Chroma 持久化后，复杂度已经高于实际需求。建议先把“可控、可解释的检索服务”做好，再决定是否需要 Agent。

## 当前检索链路

1. `app.py` 通过 Streamlit 接收用户问题，调用 `ReactAgent.execute_stream()`。
2. `ReactAgent` 使用 LangChain Agent，根据系统提示词决定是否调用 `rag_summarize` 工具。
3. `rag_summarize` 工具调用 `RagSummarizeService.rag_summarize()`。
4. `RagSummarizeService` 使用 `EnsembleRetriever` 混合 BM25 与 Chroma 向量检索，权重为 BM25 0.7、向量 0.3。
5. 检索到的 Document 拼接成 context，再交给模型总结。
6. Streamlit 页面只展示最终流式输出，不展示原始召回结果、来源、分数和命中片段。

## 发现的问题

### 1. 索引构建没有幂等性，容易重复入库

`VectorStoreService.load_document()` 中虽然定义了 `check_md5_hex()`，但实际跳过逻辑被注释掉了：

- `agent_rag_tool_project/rag/vector_store.py:54` 定义 MD5 检查。
- `agent_rag_tool_project/rag/vector_store.py:123` 到 `agent_rag_tool_project/rag/vector_store.py:125` 跳过逻辑被注释。
- `agent_rag_tool_project/rag/vector_store.py:161` 每次都会 `add_documents()`。
- `agent_rag_tool_project/rag/vector_store.py:164` 每次都会追加保存 md5。

结果是重复执行加载脚本时，同一批知识会被重复写入向量库。当前本地证据是：

- `agent_rag_tool_project/chroma_db/chunks.jsonl`：70 行。
- `agent_rag_tool_project/chroma_db/chroma.sqlite3` 的 `embeddings` 表：140 条。

这会导致 top-k 被重复内容占据，间接挤掉真正需要的相邻片段或其他文件片段。

### 2. BM25 索引与 Chroma 索引生命周期不一致

BM25 的数据来自 `chroma_db/chunks.jsonl`，向量数据来自 Chroma。两者没有统一的清理、重建、更新流程：

- `agent_rag_tool_project/rag/bm25.py:50` 每个 chunk 都以 append 方式写入 jsonl。
- `agent_rag_tool_project/rag/vector_store.py:161` Chroma 也只追加写入。
- `agent_rag_tool_project/rag/vector_store.py:169` 到 `agent_rag_tool_project/rag/vector_store.py:177` 的 `mapping2.json` 也是追加。

一旦某次加载中途失败，或者只清了其中一个索引，就会出现 BM25 和 Chroma 内容不同步。日志中也出现过 BM25 路径拼接错误导致整批加载失败，后续修复后重新加载成功的记录。

### 3. chunk_id 每次重建都会变化，评估集会快速失效

`VectorStoreService.load_document()` 使用 `uuid.uuid4()` 生成 `chunk_id`：

- `agent_rag_tool_project/rag/vector_store.py:141`

这会造成同一份文档每次重建索引后 chunk_id 都不同。当前评估集 `recall_eval.json` 依赖固定 chunk_id，因此只要重建一次索引，评估标准答案就可能失效。更稳妥的方式是用 `source_file + heading path + chunk_index + content_hash` 生成稳定 ID。

### 4. 分片策略不适合当前 Markdown 知识库

Markdown 先按标题切分，然后如果内容超过 `chunk_size` 又用 `md_spliter` 按空行切分：

- `agent_rag_tool_project/utils/file_handler.py:138` 到 `agent_rag_tool_project/utils/file_handler.py:148` 使用 `MarkdownHeaderTextSplitter`。
- `agent_rag_tool_project/rag/vector_store.py:98` 到 `agent_rag_tool_project/rag/vector_store.py:100` 长文档再次切分。
- `agent_rag_tool_project/config/chroma.yml:14` `chunk_size` 是 500。
- `agent_rag_tool_project/config/chroma.yml:21` Markdown 分隔符只有 `\n\n`。
- `agent_rag_tool_project/config/chroma.yml:23` `md_chunk_size` 是 1000。

问题是很多 Markdown 内容包含表格、mermaid、代码块、长流程说明。仅按空行切分容易出现：

- 一个语义章节被切到多个 chunk，回答需要多个片段时 top-k 不一定找全。
- 长 mermaid 或代码块占据大量上下文，但对用户问题帮助不大。
- 表格行可能和标题、解释文字分离。
- 被切分后的片段缺少父章节上下文。

### 5. 检索只返回 top-k，没有扩展上下文和父文档召回

Chroma 检索只配置了 `k: 10`：

- `agent_rag_tool_project/rag/vector_store.py:43` 到 `agent_rag_tool_project/rag/vector_store.py:45`
- `agent_rag_tool_project/config/chroma.yml:5`

BM25 固定 `k = 20`：

- `agent_rag_tool_project/rag/bm25.py:96` 到 `agent_rag_tool_project/rag/bm25.py:98`

但最终没有根据 `source_file + chunk_index` 补充前后相邻 chunk，也没有 Parent Document Retriever。对于“流程是什么”“有哪些规范”“window 有什么处理”这类需要多个片段合并的问题，单纯 top-k 很容易漏掉一部分章节。

### 6. 混合检索缺少 rerank、去重和分数调试

当前混合检索只是 `EnsembleRetriever`：

- `agent_rag_tool_project/rag/rag_service.py:40` 到 `agent_rag_tool_project/rag/rag_service.py:47`

问题：

- BM25 与向量召回结果没有统一去重策略。
- 没有暴露原始分数、排序原因、来源文件。
- 没有 rerank 过滤明显无关内容。
- BM25 权重 0.7、向量权重 0.3 是固定经验值，没有按评估集调参。

从日志看，查询“爱奇艺播放器项目中，对 window 有什么处理？”时，召回结果中混入了开发流程、智己车机连接、隐私协议等无关片段。这说明排序阶段缺少精排或元数据约束。

### 7. Agent 层会改写 query，增加不确定性

用户问题不是直接进入检索服务，而是先经过 Agent 决策。`ReactAgent` 中：

- `agent_rag_tool_project/agent/react_agent.py:11` 创建 Agent。
- `agent_rag_tool_project/agent/react_agent.py:14` 到 `agent_rag_tool_project/agent/react_agent.py:17` 注册工具。
- `agent_rag_tool_project/agent/react_agent.py:33` 到 `agent_rag_tool_project/agent/react_agent.py:40` 流式输出 Agent 状态。

系统提示词要求模型“判断是否需要调用工具”，并用“核心检索词”调用工具：

- `agent_rag_tool_project/prompts/main_prompt.txt:2`
- `agent_rag_tool_project/prompts/main_prompt.txt:16`

这会带来两个风险：

- Agent 可能不调用工具。
- Agent 可能把用户问题改写成更短的 query，丢掉限定条件。

对于小型知识库问答，建议优先采用“用户问题直接检索 + 可选 query rewrite”的确定性流程，而不是把是否检索交给 Agent。

### 8. RAG 总结链可能压缩掉已召回内容

`RagSummarizeService.rag_summarize()` 会把检索结果交给模型总结：

- `agent_rag_tool_project/rag/rag_service.py:66` 到 `agent_rag_tool_project/rag/rag_service.py:78`

即使底层召回到了多个相关 chunk，模型也可能因为“简洁准确”“聚焦提问”的提示词压缩掉一些细节。当前前端没有展示召回来源，用户看到的是总结后的答案，因此“检索结果缺失”可能包含两类情况：

- 底层确实没召回。
- 底层召回了，但总结时被省略。

建议调试时把“原始召回结果”和“最终答案”分开看。

### 9. 前端会丢失历史 assistant 完整输出

`app.py` 中页面流式展示所有 chunk，但保存历史时只保存 `response_messages[-1]`：

- `agent_rag_tool_project/app.py:25` 创建 `response_messages`。
- `agent_rag_tool_project/app.py:31` 追加 chunk。
- `agent_rag_tool_project/app.py:38` 只保存最后一个 chunk。

如果 `execute_stream()` 产生多个阶段性消息，刷新后历史里只剩最后一段。这不一定是底层检索缺失，但会造成用户在网页上看到的历史回答不完整。

### 10. 评估口径不能发现“找不全”

当前召回评估只用向量检索器，不评估线上实际的 BM25 + Chroma 混合检索：

- `agent_rag_tool_project/eval/recall/rag_recall_eval.py:20` 到 `agent_rag_tool_project/eval/recall/rag_recall_eval.py:28`

更重要的是，只要命中任意一个 gold chunk 就算命中：

- `agent_rag_tool_project/eval/recall/rag_recall_eval.py:36` 到 `agent_rag_tool_project/eval/recall/rag_recall_eval.py:37`

这会掩盖“结果缺失”。例如一个问题标准答案有 6 个 chunk，当前评估命中 1 个也算通过，但用户实际会觉得答案少内容。

### 11. 技术栈对小项目偏重

当前项目同时使用 Streamlit、LangChain Agent、Middleware、Chroma、BM25、模型总结、日志、评估集。对于“小知识库、内容不多”的场景，复杂度偏高，主要问题是：

- 问题排查链路长：前端、Agent、工具、RAG 服务、BM25、Chroma、模型总结都可能影响结果。
- 状态文件多：`chroma_db`、`chunks.jsonl`、`mapping2.json`、`md5.txt` 之间缺少统一生命周期。
- 线上回答不可解释：用户看不到命中的文件、标题、片段和分数。

## 优化方向

### 一、代码层面

1. 建立明确的索引重建命令

建议增加一个独立脚本，例如 `python -m agent_rag_tool_project.scripts.rebuild_index`，流程固定为：

- 清空 Chroma collection。
- 删除并重建 `chunks.jsonl`、`mapping2.json`、`md5.txt`。
- 扫描知识库文件。
- 生成稳定 chunk_id。
- 同步写入 Chroma 与 BM25 jsonl。
- 输出索引统计：文件数、chunk 数、每个文件 chunk 数、失败文件列表。

2. 让索引写入具备幂等性

如果保留增量更新，应使用稳定 ID，并在写入前删除同一 `source_file` 的旧 chunk。不要依赖随机 UUID 和 append-only 文件维护索引。

3. 把检索服务从 Agent 中拆出来

建议保留一个直接接口：

```python
docs = retrieval_service.search(query)
answer = answer_service.generate(query, docs)
```

网页默认走直接 RAG。只有在多工具、多步骤任务时再使用 Agent。

4. 前端增加调试面板

在 Streamlit 中展示：

- 命中文件。
- 标题路径。
- chunk_id。
- 排名。
- 检索方式：BM25 / vector / merged。
- 片段内容预览。

这样可以直接判断是“没召回”还是“召回后总结漏掉”。

5. 修正历史消息保存

`app.py` 应保存完整 assistant 输出，而不是 `response_messages[-1]`。可以把 chunk join 起来：

```python
assistant_text = "".join(response_messages)
st.session_state["message"].append({"role": "assistant", "content": assistant_text})
```

6. 清理无用依赖和未使用代码

`file_handler.py` 引入了 `UnstructuredMarkdownLoader` 但没有使用。项目也存在一些临时函数、测试入口和注释掉的示例工具。建议把实验代码和正式代码隔离，降低主链路复杂度。

### 二、检索结果层面

1. 使用“标题路径 + 正文”的结构化 chunk

每个 chunk 都应包含：

- `source_file`
- `h1/h2/h3`
- `heading_path`
- `chunk_index`
- `content_hash`
- `parent_id`
- `section_text`

对于小知识库，不需要过度切碎。优先按 Markdown 标题章节切分，章节过长时再二次切分。

2. 增加相邻 chunk 扩展

召回某个 chunk 后，根据 `source_file + chunk_index` 自动补充前后 1 个 chunk，尤其适合流程、步骤、规范类问题。

3. 增加 Parent Document Retriever

子 chunk 用于召回，父章节用于回答。这样既能保持匹配精度，也能减少回答缺少上下文的问题。

4. 对混合召回做去重和 rerank

建议流程：

1. BM25 召回 top 20。
2. Vector 召回 top 20。
3. 按 `content_hash` 或稳定 chunk_id 去重。
4. 使用轻量 rerank，或者先用规则 rerank：标题命中、文件名命中、query 关键词覆盖率。
5. 最终选 top 6 到 top 10 给模型。

小知识库可以先不用大型 rerank 模型，使用关键词覆盖率 + 标题命中加权，效果通常已经明显改善。

5. 针对中文 BM25 加分词

LangChain 的 BM25 默认对中文不一定友好。建议引入简单中文分词，例如 `jieba`，或实现自定义 preprocess_func。否则中文长句会被当成较粗粒度 token，关键词检索效果有限。

6. 引入 query rewrite 但不要交给 Agent 随机决策

可以确定性地产生 2 到 3 个查询：

- 原始问题。
- 提取关键词版本。
- 同义词/领域词扩展版本。

然后多路召回合并。比如“电脑识别不到车机”可以扩展出“adb 连接 车机 识别不到”。

7. 评估指标改成“找全率”

当前评估只看是否命中任意 gold chunk。建议增加：

- Recall@k：命中 gold chunk 的比例。
- All-hit：是否命中全部 gold chunk。
- MRR：第一个正确 chunk 的排名。
- nDCG：多个正确 chunk 的排序质量。
- Negative precision：知识库外问题是否拒答。

8. 评估线上真实链路

评估脚本应覆盖：

- vector-only
- BM25-only
- ensemble
- ensemble + rerank
- ensemble + neighbor expansion
- 最终回答是否覆盖标准答案要点

这样才能判断具体是哪一层带来收益或缺失。

## 建议的简化架构

对于当前知识库规模，可以采用更简单的架构：

```text
Streamlit
  -> RetrievalService.search(query)
      -> BM25 search
      -> Vector search
      -> merge + dedupe + rerank
      -> neighbor expansion / parent section
  -> AnswerService.generate(query, docs)
  -> show answer + sources
```

Agent 可以暂时移到可选入口，不作为普通问答的默认链路。

## 优先级建议

### P0：先解决最可能导致缺失的问题

1. 清空并重建索引，保证 Chroma 与 `chunks.jsonl` 数量一致。
2. 恢复或重做 MD5 / 稳定 ID / 按文件更新逻辑，避免重复入库。
3. 前端展示原始召回结果，区分“召回缺失”和“总结缺失”。
4. 修复 `app.py` 只保存最后一个 chunk 的问题。

### P1：提升召回完整性

1. 召回后补充相邻 chunk。
2. 增加标题路径、chunk_index、content_hash 等元数据。
3. 混合召回后去重、rerank。
4. 中文 BM25 增加分词。

### P2：提升工程可维护性

1. 拆分索引构建、检索、回答生成、前端展示。
2. 降低 Agent 在普通问答中的参与度。
3. 清理实验代码、临时函数和未使用依赖。
4. 建立稳定的召回评估与回归测试。

## 最小落地方案

如果只做一轮小改，建议按下面顺序：

1. 新增 `rebuild_index` 脚本，一键重建 Chroma、`chunks.jsonl`、`mapping2.json`。
2. chunk_id 改为稳定 ID：`md5(source_file + heading_path + chunk_index + content)`。
3. `RagSummarizeService.retriever_docs()` 返回前做去重，并按相邻 chunk 扩展上下文。
4. Streamlit 页面增加“引用来源”展示。
5. 评估脚本改为评估 ensemble，并输出每个问题缺失了哪些 gold chunk。

做完这 5 点，通常就能明显减少“网页检索结果缺失”的体感问题。
